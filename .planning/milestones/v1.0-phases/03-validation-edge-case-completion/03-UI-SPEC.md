## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #0d0d0d (--color-bg) | Page background |
| Secondary (30%) | #1c1c1c (--color-surface-2) | Input fields, modals |
| Accent (10%) | #2d6a4f (--color-primary) | Form focus states, primary buttons |
| Destructive | #e5484d (--color-danger) | Error text, validation messages |
| Text primary | #efefef (--color-text-primary) | Form labels, input text |
| Text secondary | #a8a8a8 (--color-text-secondary) | Field labels |

**Accent reserved for:**
- Input focus border and glow (3px box-shadow with `--color-primary-glow`)
- Primary button background
- Select/dropdown focus state

**Accent behavior on validation error:**
When a field has an active validation error, the accent focus ring (--color-primary) is **suppressed**:
- Error outline replaces the focus ring and uses `--color-danger` (#e5484d) instead
- Error text below the field uses `--color-danger`
- If user begins correcting the error (types/changes value), accent focus ring returns on the next focus event (or on form re-validation after keystroke)
- Example: A required field shown with danger outline and error message remains visually marked as "invalid" until the user takes action to correct it

**Destructive (--color-danger) reserved for:**
- Inline error messages below fields
- Field validation failures
- Error outlines on invalid input fields (replaces accent focus ring during error state)

**Pre-populated from:** `src/tokens.css` (lines 1-36) and existing form implementations.

## Copywriting Contract

### VALID-01: Login Field-Level Errors

**Current state:** One generic error message for all validation failures.

**Requirement:** Split into field-level inline errors shown immediately below the affected field.

| Element | Copy | Trigger |
|---------|------|---------|
| Username error | "Username wajib diisi." | Username field left blank |
| Username error (auth fail) | "Username tidak ditemukan." | After form submission, `store.cariAkun()` returns null for username |
| Password error | "Password wajib diisi." | Password field left blank |
| Password error (auth fail) | "Password salah untuk akun ini." | After form submission, `store.cariAkun()` returns null due to password mismatch |
| Role missing error | "Pilih role terlebih dahulu." | Role not selected and form submitted |
| Generic fallback | "Gagal masuk. Periksa kembali username, password, dan role Anda." | Unexpected error state |

**Rendering pattern:**
- Same as `InputData.jsx` (line 281, 296-298, 317-319): `<p style={errorStyle}>{errors.field}</p>` below each field
- Error color: `--color-danger` (#e5484d)
- Font size: 0.85rem
- Margin: 6px 0 0
- Line height: 1.5

**Clear-on-change behavior:**
- Errors cleared when user types in either field (as currently done in Login.jsx lines 189-190, 211-212)
- Errors re-evaluated on form submission

### VALID-02: StatusDistribusi Armada & ETA Validation

**Current state:** Armada and ETA fields accept blank values when status is "dalam-pengiriman".

**Requirement:** Both fields are required when status is "dalam-pengiriman" and must show inline errors if left blank.

| Element | Copy | Trigger |
|---------|------|---------|
| Armada required error | "Armada / Sopir wajib diisi saat status Dalam Pengiriman." | Status set to "dalam-pengiriman" and armada field left blank on save |
| ETA required error | "Estimasi Tiba (ETA) wajib dipilih saat status Dalam Pengiriman." | Status set to "dalam-pengiriman" and ETA field left blank on save |

**Rendering pattern:**
- Same as `InputData.jsx` errorStyle (line 228-233)
- Error color: `--color-danger`
- Font size: 0.85rem
- Margin: 6px 0 0
- Line height: 1.5
- Displayed inline below each field in the modal

**Validation timing:**
- On "Simpan Status" button click, before calling `store.updateKeputusan()`
- If errors exist, prevent save and display errors
- Clear errors only when user types in the affected field

**Button state:**
- "Simpan Status" button remains enabled (not disabled) during validation
- On validation failure, errors render and focus does not change

### VALID-03: InputData Empty City List Message

**Current state:** Dropdown presents `<option value="">⚑ Pilih kota</option>` with no explanation if daftarKota is empty, leading to confusion.

**Requirement:** When no cities are configured, show a clear explanatory message instead of a silent empty dropdown, and prevent form submission.

| Element | Copy | Condition |
|---------|------|-----------|
| Empty state message | "Belum ada kota yang dikonfigurasi. Hubungi Admin untuk menambahkan kota terlebih dahulu." | `daftarKota.length === 0` |
| Error message on field | "Pilih kota terlebih dahulu." | `daftarKota.length > 0` but form.kota is empty |

**Rendering pattern (empty state):**
- Display a `<p>` or `<div>` styled like existing form guidance (color: `--color-text-muted`, font-size: `--text-sm`)
- Positioned where the select dropdown would be
- Same as card/form section padding context

**Rendering pattern (field-level error):**
- Reuse existing `errorStyle` from line 228-233 if user attempts submit with blank city despite list existing

**Validation logic:**
- On component render: Check if `daftarKota.length === 0`
  - If true: Render empty state message, do not render the select dropdown
  - If false: Render select dropdown normally
- On form submission: If no cities exist, prevent submission with a toast message: "Belum ada kota yang dikonfigurasi. Hubungi Admin untuk menambahkan kota terlebih dahulu."

## Implementation Notes

### Reusable Patterns (Existing Code Reference)

All validation UIs in Phase 3 follow the **InputData.jsx pattern** (VALID-03 reference):

**Error rendering pattern:**
```jsx
const errorStyle = {
  margin: "6px 0 0",
  color: "var(--color-danger)",
  fontSize: "0.85rem",
  lineHeight: 1.5,
};

// ... in JSX:
{errors.fieldName ? <p style={errorStyle}>{errors.fieldName}</p> : null}
```

**Validation state pattern:**
- Errors stored in `errors` object (key = field name, value = error message string)
- Validation runs on form submission or on field change (re-validate on every keystroke to clear old errors)
- `validate(form)` or `validate()` function returns error object; empty object = valid

**Clear-on-change pattern:**
- On field change: `setErrors({})` or revalidate immediately to show/hide errors
- On successful submission: `setErrors({})` to clear all errors

**Modal & dropdown pattern (StatusDistribusi):**
- Modals use the same inline error rendering below affected fields
- Errors persist until user edits the field or submits
- Button click (save/submit) triggers validation before mutation

### Design Token Constants

All colors, sizes, spacing, and fonts are CSS custom properties defined in `src/tokens.css`. Never hardcode values:

- Colors: `var(--color-*)`
- Spacing: `var(--space-*)`
- Typography: `var(--text-*), var(--font-*), var(--font-weight-*), var(--leading-*)`
- Shadows & transitions: `var(--shadow-*), var(--transition-*)`

### No UI Changes Required for VALID-04

New accounts created via Register (Phase 3 requirement VALID-04) must use the `getNextId` convention instead of `Date.now()`. This is a **backend store convention fix only** — no UI surface or user-visible change. Excluded from this UI-SPEC per project constraint. Implementation is in `src/pages/Register.jsx` line 99, changing `id: \`U${Date.now()}\`` to `id: store.getNextId(store.getDaftarAkun(), 'U')` or similar.
