# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **Framing for this phase (read first):** This is a *consistency port*, not a redesign. Landing, Login, and Register must end up using the same components, tokens, and visual language already established by `Dashboard.jsx`, `ManajemenKota.jsx`, and the rest of the authenticated app. Every value below is extracted from the existing codebase (`src/tokens.css`, `src/components/Tombol.jsx`, `src/components/Card.jsx`, `src/components/IkonDaun.jsx`, `src/components/auth/AuthShared.jsx`) — nothing here is a new visual direction. Where the existing Landing/Login/Register visuals cannot be expressed with current shared components without modifying those components' shared styling, this is flagged explicitly in the **Shared-Component Risk Register** section, because `Tombol`/`Card`/`IkonDaun` are consumed by every other page in the app.

## Spacing Scale

Declared values — this is the **existing** `--space-*` scale from `src/tokens.css`, already a clean 4px-multiple system. Use these tokens, not literal pixel values, in all new/modified JSX:

| Token | Value | Usage |
|-------|-------|-------|
| `--space-1` | 4px | Icon-to-text gaps, tight inline spacing |
| `--space-2` | 8px | Compact element spacing (badge padding, small gaps) |
| `--space-3` | 12px | Default internal gaps (form field internal spacing) |
| `--space-4` | 16px | Default element spacing (form field stacks — matches current Login/Register `gap: var(--space-4)`) |
| `--space-5` | 20px | Card/section internal padding |
| `--space-6` | 24px | Card padding (`Card` component default is `var(--space-6)`), modal section breaks |
| `--space-8` | 32px | Auth card top-of-form spacing (current Login/Register use this above the form heading block) |
| `--space-10` | 40px | Auth card outer padding (current Login/Register form `padding`) |
| `--space-12` | 48px | Landing section horizontal padding (current Landing header/section padding uses literal `48px` — equivalent to `--space-12`, should be expressed as the token during the port) |
| `--space-16` | 64px | Landing major section vertical rhythm where currently hardcoded as `120px`/`100px`/`80px` literals — see Exceptions below |

**Exceptions:**

- Landing's hero/section vertical padding currently uses literal values larger than the declared scale (`120px`, `100px`, `80px`, `64px` for mobile). These do not map cleanly onto `--space-*` (max declared token is `--space-16` = 64px). **Decision: keep these specific literal large-spacing values as page-level exceptions** — they are Landing-specific marketing-page rhythm, not a token gap to fix. Do not invent new `--space-20`/`--space-24` tokens for this phase (scope is component reuse, not token-system expansion). Express them as literals exactly as today; only refactor spacing that maps onto an existing token (e.g. `48px` → `var(--space-12)`, `24px` → `var(--space-6)`, `16px` → `var(--space-4)`).
- Auth modal touch targets (role pills, close button, eye-icon toggle) keep their current sizes (`32px` close button, full-width role pills) — these are already token-driven via `AuthShared.jsx` and `tokens.css` `.role-pill`/`.auth-close-btn` classes; no change needed.

| Display (Landing hero `h1`, big stat numbers) | `clamp(...)` fluid, existing Landing hero already uses `clamp(1.9rem, 8vw, 5rem)` — keep as-is, this is a marketing-page exception already established in the codebase | `var(--font-weight-bold)` (700) / 800 for hero stat numbers (existing literal `fontWeight: 800` in `STATISTIK_LIST` rendering — acceptable, matches `--font-weight-bold` family intent) | 1.08–1.2 |

### Dimension 4 policy exception (requested, justified — not a silent overage)

**Declared scale exceeds the standard policy max (4 sizes / 2 weights).** This phase declares up to 6 distinct `--text-*` tokens in active use (`--text-2xs`, `--text-xs`, `--text-sm`/`--text-base`, `--text-xl`, `--text-2xl`, plus the Landing-only `clamp()` display size) and 3 weight tokens (400, 600, 700) plus one literal exception (800 for stat-number emphasis). This is requested as an explicit, justified exception to the Dimension 4 standard policy, for the following reasons:

1. **The full scale is not introduced by this phase — it is the pre-existing, already-shipped app-wide scale.** `src/tokens.css` declares 10 size tokens (`--text-2xs` through `--text-4xl`) and 4 weight tokens; this predates Phase 4 entirely and is the single shared type system for every page in the app.
2. **The broad range is already in active use across authenticated pages that are out of scope for this phase to change**, confirmed by direct inspection:
   - `src/components/MetricCard.jsx` (used by `Dashboard.jsx` and other KPI surfaces): `--text-2xs`, `--text-xs`, `--text-2xl`, plus a `valueFontSize` override prop — 3+ sizes in one shared component.
   - `src/pages/Dashboard.jsx`: uses `--text-2xs`, `--text-xs`, `--text-sm`, `--text-xl`, `--text-2xl`, `--text-3xl` — 6 distinct sizes — plus weights 600/700 via `--font-weight-semibold`/`--font-weight-bold`.
   - `src/pages/KeputusanDistribusi.jsx` (line 278): uses literal `fontWeight: 800` for an emphasized ranking/score number — the exact same literal-800 pattern this phase's Landing hero stats use. This is not a Landing-only precedent; it already exists in a shipped, in-scope-elsewhere authenticated page.
   - `src/pages/AnalisisRanking.jsx`: literal `fontWeight: 700` for ranking emphasis, plus `--text-sm` for body/labels.
   - `src/components/Layout.jsx`, `src/pages/InputData.jsx`, `src/components/Tabel.jsx`, `src/pages/Login.jsx`, `src/pages/Register.jsx`: all consume multiple `--text-*` tokens and the semibold/bold weight tokens as part of normal, already-shipped UI (nav items, table headers, form labels, auth headings).
3. **Trimming Landing/Login/Register's declared scale to ≤4 sizes / ≤2 weights would make this phase's pages LESS consistent with the rest of the app**, which directly contradicts this phase's stated goal (DESIGN-01: "Landing, Login, and Register must end up using the same components, tokens, and visual language already established by `Dashboard.jsx`... and the rest of the authenticated app"). Forcing Landing's hero/stat numbers onto a narrower scale than `Dashboard.jsx`/`MetricCard.jsx`/`KeputusanDistribusi.jsx` already use would create a new, artificial inconsistency rather than removing one.
4. **Scope boundary:** Per `CLAUDE.md`, this phase's scope is "completion and polish of existing functionality... no new pages, roles, or business domains" and the UI-SPEC framing above states this is "a consistency port, not a redesign." Reducing or restructuring the shared `tokens.css` type scale itself is a token-system redesign, which is explicitly out of scope for this phase (see Spacing Scale "Exceptions" section, which makes the same call for spacing literals).

**What this phase does NOT do (the actual commitment, scoped correctly):**
- It does not introduce any `--text-*` or weight value that doesn't already exist in `src/tokens.css` or in an already-shipped page's literal usage (700/800 emphasis numbers).
- It does not expand the *range* of sizes/weights used specifically by Landing/Login/Register beyond what Landing already used pre-port (no new sizes are being added to Landing as part of this rebuild — the same headline/stat/body/label sizes already in `Landing.jsx` today are preserved, just re-rendered through shared components where applicable per the Risk Register above).
- Login/Register's own type usage stays narrow and unchanged: body (`--text-sm`/`--text-base`, 400), label (`--text-xs`, 600), heading (`--text-2xl`, 700) — i.e. 3 sizes / 2 weights, well within standard policy on those two pages specifically. The exception is driven by Landing's hero/stats section and by aligning with the wider app scale, not by Login/Register individually.

**Requested sign-off:** Approve Dimension 4 for this phase under this documented exception (full existing `tokens.css` scale + the literal-800 stat-emphasis precedent already shipped in `KeputusanDistribusi.jsx`), rather than requiring Landing/Dashboard-aligned pages to adopt a scale narrower than the rest of the app.

**Font family mapping (already established, keep consistent):**
- `var(--font-display)` — headings, brand wordmark, numeric KPI values
- `var(--font-body)` — body copy, form inputs, buttons, nav links
- `var(--font-mono)` — numeric/stat values rendered in mono in Landing's `MockupDashboard`/`FiturVisual` (this is consistent with `Dashboard.jsx`'s `formatTonase`/KPI rendering elsewhere — keep)

**Auth modal heading hierarchy (Login/Register, already correct, do not change):** brand row (`IkonDaun` + "Switera" wordmark at `var(--text-lg)` semibold) → `h1` page title at `var(--text-2xl)` bold ("Masuk" / "Buat Akun") → subtitle at `var(--text-sm)` muted. This hierarchy is already token-driven in current Login/Register and should be preserved exactly.

## Shared-Component Risk Register

**This section is the load-bearing output of this UI-SPEC for Phase 4.** `Tombol`, `Card`, and `IkonDaun` are consumed by every authenticated page in the app (`Dashboard`, `ManajemenKota`, `ManajemenData`, `InputData`, `StatusDistribusi`, `KeputusanDistribusi`, `Laporan`, `AnalisisRanking`, `RiwayatAktivitas`, `Layout`). Any change to these components' own default styling (their CSS classes in `tokens.css`, or their default prop values) propagates to all of those pages. DESIGN-04 explicitly requires zero regression on all of them.

### Risk 1 — `Tombol` variant system does not cover Landing's button visual language

**Finding:** `Tombol` ships exactly 3 variants — `tombol-primer` (filled `--color-primary` green), `tombol-sekunder` (transparent, bordered, muted text), `tombol-bahaya` (filled `--color-danger` red). Landing's current `LandingButton` ships a *different* 2-variant system already defined in `tokens.css` under separate classes — `landing-btn-primer` (white fill, black text — inverted/high-contrast marketing style) and `landing-btn-ghost` (fully transparent, muted white text, no border). These are visually and semantically distinct from `Tombol`'s variants; `landing-btn-primer` in particular has **no equivalent** in `Tombol` (white-on-black inversion is a marketing-page-only treatment, never used in the authenticated app shell).

**Decision (locked, not discretionary):** Do **not** add a 4th variant to `Tombol` to absorb Landing's white-inverted button style, and do not modify `tombol-primer`'s existing color/shadow to "fit" Landing. Doing so would change the default look of every primary button across the entire authenticated app (Dashboard's "+ Tambah Permintaan", ManajemenKota's "+ Tambah Kota", every form submit button, etc.) — an unacceptable regression surface for a marketing-page-only need.

**Resolution path for the executor:** Two acceptable options, in priority order:
1. **Preferred:** Use `Tombol` with its existing `variant="primer"` (green fill) for Landing's primary CTAs ("Mulai Sekarang", "Daftar", "Daftar Sekarang") instead of the current white-inverted treatment. This is a deliberate, scoped visual adjustment to Landing's button color **only** — it does NOT touch `Tombol`'s shared styling, and it brings Landing into closer alignment with the rest of the app's color language (which is the actual intent of DESIGN-01: "feel like the same product as the rest of the app"). The white-inverted look is itself part of the inconsistency being fixed.
2. **Acceptable fallback if op visual fidelity to current Landing screenshot matters more:** Keep Landing's nav-area "Masuk"/"Daftar" buttons and hero CTAs visually as today (using the existing `landing-btn-*` CSS classes, unchanged, scoped to Landing only) BUT render them through a thin wrapper that still imports and composes `Tombol`'s ripple-handling behavior pattern, or accept that `LandingButton` remains a Landing-local presentational component that is NOT one of the three components named in the success criteria (Tombol/Card/IkonDaun) — in which case it must be clearly justified as "Landing's CTA buttons are intentionally styled distinctly for marketing-page contrast, consistent with existing `landing-btn-*` tokens.css classes that predate this phase." This path satisfies "renders using design tokens" (success criterion 1 says "Tombol, Card, IkonDaun, **and design tokens**" — tokens are already used) but is weaker on literal `Tombol` adoption.

**Recommendation: Option 1.** Replacing Landing's hero/nav CTAs with actual `Tombol variant="primer"` / `variant="sekunder"` (for "Masuk"/ghost-style nav buttons) is the only path that genuinely satisfies "renders using `Tombol`" rather than "renders near `Tombol`." The visual delta (green fill vs. white fill on the hero buttons) is a minor, on-brand change to Landing only, and does not touch any shared component file.

**Files affected if Option 1 is taken:** Only `Landing.jsx` (replace `LandingButton`/`SIZE_STYLES` local component with direct `Tombol` usage). Zero changes to `src/components/Tombol.jsx` or `tokens.css`'s `.tombol-*` classes. The existing `.landing-btn-*` CSS classes in `tokens.css` become dead code after the port and may be removed once Landing no longer references them (executor's call — removing unused CSS is safe and improves AUDIT-01 readiness, but is not required for DESIGN-01 to pass).

### Risk 2 — `Card`'s default padding/background does not match Landing's bespoke surfaces

**Finding:** `Card` renders `var(--color-surface)` background, `var(--radius-lg)` corners, `var(--space-6)` padding, with optional `accent` top-border and `hoverable`/`shimmer` props. Landing's bespoke surfaces (`VisualCardShell`, `MockupDashboard`, FAQ accordion container, testimonial chips) use *different* combinations: `var(--color-surface)` background ✓ matches, but custom `border: 1px solid var(--color-border-mid)` (not `Card`'s default `var(--color-border)`), custom `padding: 28px` (not a `--space-*` token-aligned value — closest is between `--space-6` (24px) and `--space-8` (32px)), and bespoke `boxShadow` values (`0 20px 60px rgba(0,0,0,0.4)`) that `Card` does not natively support as a prop.

**Decision (locked):** Do **not** add new props to `Card` (e.g. a `boxShadow` prop or a `border-mid` variant) solely to accommodate Landing's bespoke shadow treatment — this is a Landing-specific visual flourish (large soft shadow for marketing depth), not a need shared by any authenticated page. Any prop addition to `Card` must be additive-only (new optional prop, default `undefined`/off) so existing consumers (`Dashboard`, `ManajemenKota`, etc.) get zero visual change — if the executor determines a prop addition is genuinely necessary, it must default to current `Card` behavior exactly and only Landing opts in.

**Resolution path for the executor:** Wrap `Card`'s `style` override prop (already supported — `Card` accepts and merges a `style` prop, see `src/components/Card.jsx:8,28`) to layer Landing-specific shadow/padding/border on top of `Card`'s defaults, exactly the same pattern already used elsewhere in the app (e.g. `Dashboard.jsx:390` `<Card style={{ minHeight: "420px" }}>`). This satisfies "renders using `Card`" without modifying `Card.jsx` or its CSS class `.app-card` in `tokens.css`. Apply `Card` to: `VisualCardShell` (feature visual containers), the FAQ section container, and the testimonial-chip container if a card-like surface is appropriate — NOT to fully bespoke decorative elements like the hero glow gradient or the mockup dashboard chrome (those remain custom, as they have no `Card`-shaped semantic meaning).

**Files affected if this path is taken:** Only `Landing.jsx`. Zero changes to `src/components/Card.jsx` or `tokens.css`'s `.app-card` class.

### Risk 3 — `IkonDaun` duplication (the clearest, lowest-risk fix)

**Finding:** `Landing.jsx` (lines 133–157) defines a **local, duplicate copy** of `IkonDaun` instead of importing the shared `src/components/IkonDaun.jsx`. The two implementations are byte-for-byte identical (same `viewBox`, same path data, same default props). This is pure duplication with zero behavioral difference — the lowest-risk, highest-confidence fix in this phase.

**Decision:** Delete the local `IkonDaun` function from `Landing.jsx` and import the shared one (`import IkonDaun from "../components/IkonDaun";`), exactly as `Login.jsx` and `Register.jsx` already do. Zero visual change (the implementations are identical). Zero risk to other pages — `IkonDaun.jsx` itself is untouched.

### Risk 4 — Login/Register's `auth-input` and form-row patterns vs. app-wide form conventions

**Finding:** Login/Register already use `AuthShared.jsx`'s `inputBaseStyle`/`fieldLabelStyle`/`FieldIcon`/`ErrorText`/`RolePills` — these were extracted as shared auth pieces in Phase 3. They are visually consistent with the app's other form inputs (`InputData.jsx`, `ManajemenKota.jsx` use similar bordered/`--color-surface-2` input styling, though not via a literally shared `<Input>` component — the app has no generic shared `Input` component to date). **This is acceptable and not a regression risk**, since `AuthShared.jsx` is scoped to auth pages only and isn't consumed elsewhere. No action required beyond what Login/Register success criteria (DESIGN-02/03) already specify: continue using these already-shared pieces, do not regress them.

**Remaining gap for DESIGN-02/03:** Login/Register's outer modal chrome (the `position: fixed` overlay, the `backdropFilter` glass card, the background image layer) and the submit/link buttons (`.auth-submit-btn`, `.auth-link-btn` CSS classes) are still bespoke — not built on `Tombol` or `Modal`. The phase goal says "renders using the shared component library instead of inline styles" (not "must use `Tombol` literally" — unlike Landing's success criterion 1, Login/Register's criteria 2/3 don't name specific components). **Recommendation:** Replace `.auth-submit-btn` usage with `Tombol variant="primer"` and `.auth-link-btn` with a plain inline text-button pattern already used elsewhere (e.g. `RiwayatAktivitas.jsx`'s `link-underline-hover` class), and consider whether `Modal.jsx`'s overlay/dialog pattern (focus trap, Escape-to-close, ARIA dialog semantics — see `src/components/Modal.jsx`) should be reused for the auth overlay shell instead of Login/Register's own hand-rolled `position: fixed` overlay, **provided** the background-image/glass treatment that makes the auth modal visually distinct from generic app modals can be preserved via `Modal`'s existing support for custom content (it has no `className`/`style` override on the outer dialog today — adding one, additive-only/optional, would be a one-line, zero-regression change to `Modal.jsx` since no existing consumer passes such a prop). Flag this `Modal.jsx` change (if taken) for explicit review since `Modal` is shared — but it is a strictly additive optional-prop change, the safest category of shared-component modification.
