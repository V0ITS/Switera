  <action>
    Create server/src/middleware/validate.js exporting a factory `validate(schema)` that returns Express middleware. It parses `req.body` with the given Zod schema via `schema.safeParse(req.body)`. On failure, respond 400 with a field-level error object: map Zod's `error.issues` into `{ error: "Validasi gagal.", fields: { <path>: <message>, ... } }` where each key is the dotted field path (issue.path joined by ".") and the value is issue.message. On success, replace `req.body` with the parsed/validated data (`req.body = result.data`) so downstream handlers receive only allowlisted fields — this is the mass-assignment defense (see threat_model T-08-MASSASSIGN): handlers must never blind-spread an unvalidated req.body into Prisma.

    Write Zod issue messages in Indonesian where the schema authors (Wave 2/3) supply custom messages; validate.js itself only needs the generic top-level "Validasi gagal." label plus per-field passthrough of whatever message the schema defines.

    Create server/src/middleware/errorHandler.js exporting a 4-arg Express error-handling middleware `(err, req, res, next)`. Map known service-layer errors to HTTP status by inspecting err.message against the established Indonesian error strings:
    - message contains "sudah ada" or "sudah digunakan" -> 409 (conflict / duplicate)
    - message contains "tidak bisa dihapus" -> 409 (referential block)
    - message contains "tidak ditemukan" -> 404
    - a custom conflict error used by the race-safe approval path (08-03) -> 409 (detect via an `err.statusCode` property if present; prefer reading err.statusCode when set, falling back to message matching)
    - everything else -> 500 with a generic `{ error: "Terjadi kesalahan pada server." }` and NEVER echo err.stack or the raw err.message for the 500 case (T-08-ERRLEAK). For the mapped 4xx cases, the Indonesian err.message IS the user-facing message and may be returned.

    Prefer an `err.statusCode` convention: if a thrown error sets `err.statusCode`, honor it directly. Document this so Wave 2/3 services can throw `Object.assign(new Error(msg), { statusCode: 409 })` for precise control. Keep the message-matching as a fallback for the Phase 6 service errors that pre-date this convention.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser (Vite 5173) -> Express API | Untrusted request bodies and headers cross here; CORS governs which origins the browser permits |
| route handler -> Prisma | Validated DTO crosses into the data layer; must be allowlisted, never raw req.body |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-MASSASSIGN | Tampering / Elevation | validate.js -> all Wave 2/3 CRUD routes | mitigate | validate() replaces req.body with `schema.parse` output (allowlist); handlers spread only the validated DTO into Prisma, never blind `...req.body` |
| T-08-ERRLEAK | Information Disclosure | errorHandler.js | mitigate | 500 path returns a generic Indonesian message and never echoes err.stack or raw err.message; only mapped 4xx Indonesian messages are surfaced |
| T-08-CORS | Spoofing / CSRF surface | index.js cors config | mitigate | CORS origin locked to the single Vite dev origin (no wildcard); not `origin: true` reflective |
| T-08-SQLI | Injection | stokService.js / all services | accept | Prisma parameterizes every query; no raw SQL (`$queryRaw`/`$executeRaw`) is used anywhere in this phase — verified non-issue, no string interpolation of city-name/free-text into SQL |
</threat_model>
