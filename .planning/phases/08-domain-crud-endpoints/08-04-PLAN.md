<objective>
Port the Keputusan (distribution decision) domain to a real REST API (API-01/API-02) and close LOGIC-02 — the headline race condition in this phase. `updateKeputusan` in src/store.js (line 453) is a check-then-act flow with zero concurrency guard: it reads `existing.status`, decides whether the status changed, then unconditionally overwrites the row. Two concurrent approval requests for the same decision can both read the pre-change status and both believe they are the one transitioning it, double-firing notifications/activity and potentially double-allocating the same city's stock.

LOGIC-02 race-safety decision (binding for this plan, not a TODO): **Optimistic locking via a conditional `updateMany` WHERE-clause guard**, not a DB unique constraint and not a `$transaction` with manual row locking.

Rationale:
- A unique constraint (e.g. partial unique index on `kotaTujuanNama + status='dalam-pengiriman'`) was considered and REJECTED: the data model allows multiple Keputusan rows per city across different statuses/time periods (see seed data: KPT-001/002/003 are 3 different cities, but nothing in the domain says only one decision can ever exist per city — `kota_tujuan` is not a uniqueness boundary here, the DECISION ROW's status transition is). A unique constraint would either be too narrow (doesn't model the real contested resource) or would incorrectly block legitimate sequential decisions for the same city.
- A `prisma.$transaction` with a manual read-then-write still has the SAME TOCTOU race unless it uses `SELECT ... FOR UPDATE` (row locking) or Serializable isolation — that is real engineering complexity disproportionate to this school-project scale (REQUIREMENTS.md "Out of Scope": "Pessimistic row-locking across the whole API").
- The correct minimal-complexity fix: make the UPDATE itself the lock. `prisma.keputusan.updateMany({ where: { id, status: existingStatusReadJustBefore }, data: {...} })` is a single atomic SQL `UPDATE ... WHERE id = ? AND status = ?` statement — Postgres guarantees only one of two concurrent identical statements can match and update the row; the loser's `result.count` is `0`. This requires no explicit transaction or row lock and is the standard optimistic-concurrency idiom — exactly proportionate to "one race-prone mutation" (REQUIREMENTS.md "Out of Scope" rationale for rejecting pessimistic locking).

Purpose: Decision approval is the one mutation in this entire codebase with a real, documented concurrency bug. This plan is the fix.
Output: keputusanService.js (race-safe updateKeputusan), keputusanSchemas.js, keputusanRoutes.js, a concurrent-request verify script proving exactly-one-success, router mounted.
</objective>
