<objective>
Port the distribution-ranking/recommendation engine (`computeRekomendasiDistribusi`) and the KPI metrics calculation (`computeKpiMetrics`) from `src/utils/distribusi.js` to a server-side endpoint, satisfying LOGIC-01: the calculation must run server-side, reading live data from the database at request time — never a cached or client-supplied value.

`src/utils/distribusi.js` is pure, framework-agnostic JS with zero React/store coupling (per STATE.md's patterns-established note: "the most testable units in the codebase") — the algorithm itself (`aggregatePermintaanRanking`, the scoring formula, the allocation walk) ports verbatim, unchanged. The ONLY change is the data source: instead of receiving `permintaan`/`daftarKota`/`stokTbs` as in-memory arrays passed in by a React component, the server-side version fetches them fresh from the database via the existing kota/permintaan/stok services on every single request — no caching, no memoization across requests, no module-level snapshot taken at server-start.

Purpose: This is the success criterion most likely to be silently violated by a naive port (e.g. computing once at startup, or caching the daftarKota/stokTbs lookups) — the plan makes the "fresh read every call" requirement an explicit, testable acceptance criterion, not an assumption.
Output: distribusiService.js (verbatim algorithm + live DB reads), distribusiRoutes.js, a verify script that mutates the DB directly between two calls and asserts the second call reflects the change.
</objective>

src/utils/distribusi.js exports (PURE functions, copy the logic verbatim — do not "improve" or refactor the algorithm itself):
- `aggregatePermintaanRanking(permintaan)` -> ranking array grouped by kota, sorted by totalPermintaan desc then earliest tanggal_input
- `computeRekomendasiDistribusi(permintaan, daftarKota, stokTbs)` -> the scoring + allocation walk (0.65 demand-weight + 0.35 capacity-weight, then greedy allocation against remaining stokTbs in score order) -> array of { kota, totalPermintaan, kapasitas, skor, alokasi, terpenuhiPenuh, dibatasiKapasitas }
- `computeKpiMetrics(keputusan, permintaan, daftarKota)` -> { fulfillmentRate, onTimeRate, avgSiklusJam, kotaTercover, totalKota }

These three functions take plain arrays/numbers as arguments — they have NO knowledge of where the data comes from. The server-side service's job is ONLY to fetch fresh arrays and call them; the math itself does not change.

Existing services to call for fresh data (from 08-01/08-02/08-03, all already return store.js-shaped snake_case arrays/numbers):
- `kotaService.getDaftarKota()` -> Kota[] with { nama, kapasitas } (already this shape, Phase 6)
- `permintaanService.getPermintaan()` -> Permintaan[] snake_case shape { kota, tanggal_permintaan, tanggal_input, jumlah_permintaan, ... } (08-03)
- `stokService.getStokTbs()` -> number (08-01)
- For KPI: also need `keputusanService.getKeputusan()` -> Keputusan[] snake_case shape { kota_tujuan, volume_tbs, status, waktu_menunggu, waktu_selesai, eta? } (08-04) — NOTE: store.js's computeKpiMetrics reads `item.eta` on keputusan entries, but neither src/data/keputusan.json nor the Prisma Keputusan model has an `eta` field (confirmed absent from schema.prisma) — this is a pre-existing latent gap in the v1 data model, not something this plan introduces. Port the function exactly as written (the `if (!item.eta) return true` branch will simply always be true since `eta` is always undefined server-side, identical behavior to the client today, since the client's keputusan.json also never sets `eta`). Do not invent an eta field.
