<objective>
Close LOGIC-03 IN FULL: activity-log entries and notifications must be generated server-side as part of the SAME mutation request that triggers them, never requiring a separate client call — for every mutation that src/store.js instruments today, not only permintaan/keputusan. This plan does two things: (1) ports the Notifikasi and ActivityLog domains to real REST read/write endpoints (completing API-01's full 7-domain coverage), and (2) wires the generation hooks directly into the permintaan, keputusan, kota, AND stok services built in 08-02/08-03/08-04, so e.g. `addPermintaan`, `updateKeputusan`, `tambahKota`, `updateKota`, `hapusKota`, and `setStokTbs` each push their activity-log row (and, for permintaan/keputusan, a notification row) before they return — not as a separate endpoint the frontend would have to call afterward.

src/store.js's `recordActivity` is called inside ALL of: addPermintaan, updatePermintaan, removePermintaan, addKeputusan, updateKeputusan, removeKeputusan, restoreKeputusan, tambahKota, updateKota, hapusKota, and setStokTbs. The first seven are wired by this plan's permintaan/keputusan side; the last four (tambahKota/updateKota/hapusKota/setStokTbs) are wired by Task 2's kota/stok extension below — closing the full scope, not a permintaan/keputusan-only subset.

This requires one schema addition: ActivityLog currently has no `aktor`/`role` source server-side (store.js derives these from `state.userAktif`/`state.roleAktif`, which only exist client-side). Server-side mutations have `req.user` (from requireAuth: `{ id, username, role }`) instead — the route layer passes `req.user.username` (mapped to a display name) and `req.user.role` into the service call so the activity log still records "who did this," using the authenticated identity rather than a client-trusted field.

Purpose: This is the requirement most likely to be silently violated by treating notifikasi/activityLog as "just another CRUD domain" — the plan makes explicit that the generation must live INSIDE the permintaan/keputusan/kota/stok service functions themselves, not in a separate route the frontend calls second, and that EVERY store.js mutation with a recordActivity call site gets its server-side equivalent, not just the two domains with the richest side effects.
Output: notifikasiService.js, activityLogService.js, their routers, and modifications to permintaanService.addPermintaan + keputusanService.updateKeputusan (and addKeputusan/removeKeputusan/restoreKeputusan) plus kotaService.tambahKota/updateKota/hapusKota and stokService.setStokTbs to fire the same notification/activity side effects store.js fires today, inside the same function call. Also removes the Phase 7 throwaway /protected demo router, now that real domain routes establish the pattern.
