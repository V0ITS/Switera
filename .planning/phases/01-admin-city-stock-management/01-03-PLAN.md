## Phase Goal (User Story)

**As an** Admin, **I want to** manage the city list and TBS stock level through a dedicated page, **so that** city data and stock stay accurate and propagate everywhere without any silent data-integrity gaps.

## Decision note (Assumption A1, confirmed for this plan)

The D-01 block-delete check counts references in **active `state.keputusan` only** (not the cancelled-only entries that may exist solely in `state.riwayatKeputusan`). Rationale: blocking deletion on a historical cancelled decision would make a once-decided-then-cancelled city undeletable forever. The cascade-RENAME (D-02), by contrast, updates ALL THREE collections including `riwayatKeputusan`, because stale history must never show an outdated name.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Page delete request → store mutators | Admin-initiated delete/rename crosses into cross-collection state mutations |
| Store mutation → referencing collections | A single rename/delete must keep permintaan/keputusan/riwayatKeputusan consistent |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering (data integrity) | Rename leaving stale references in keputusan/riwayatKeputusan | mitigate | `updateKota` cascade rewrites all three collections (permintaan.kota, keputusan.kota_tujuan, riwayatKeputusan.kota_tujuan) in one mutation, single notify(). |
| T-03-02 | Tampering (dangling reference) | Deleting a city still referenced, orphaning permintaan/keputusan | mitigate | `hapusKota` throws when getKotaReferenceCounts is non-zero; page blocks delete BEFORE calling it. No code path deletes a referenced city (D-01). |
