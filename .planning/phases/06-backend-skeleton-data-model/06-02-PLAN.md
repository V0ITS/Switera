<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------|
| Seed script -> database | Trusted, one-time/dev-repeatable data load; not exposed to untrusted input (no HTTP boundary in this plan) |
| Plaintext seed passwords (in seed.js source) -> bcrypt hash (in DB) | The only point in DATA-02 where a plaintext value exists at all, and only transiently in process memory, never persisted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-06-04 | Information Disclosure | `server/prisma/seed.js` (contains the 3 known plaintext seed passwords as source literals) | accept | These are publicly-known school-project demo credentials already visible in `src/store.js` today (committed to this same git repo); no new exposure introduced. Real account creation in Phase 7 (AUTH-02) will not use this path. |
| T-06-05 | Tampering | Seed script re-run against a non-empty database | mitigate | `upsert()` instead of `create()` makes reseeding idempotent — re-running cannot duplicate rows or silently fail mid-script on the 2nd+ row |
| T-06-06 | Repudiation | bcrypt cost factor too low for production | accept | Cost factor 10 is the bcryptjs library default and adequate for a school-project scale (3 accounts, local dev/demo use, not internet-facing production) |
</threat_model>
