## Trust Boundaries

| Boundary | Description |
|----------|--------|
| Service-layer function calls -> Prisma -> PostgreSQL | Trusted internal boundary for this phase (no HTTP/network input yet — Phase 8 adds the untrusted-input boundary at the route layer) |
| `prisma.$transaction` cascade-rename -> partial-failure state | A multi-table write that must be all-or-nothing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|------------------|
| T-06-07 | Tampering | `updateKota` cascade-rename across 4 tables | mitigate | All 4 writes (Kota + 3 referencing tables) execute inside one `prisma.$transaction([...])` — a failure on any step rolls back the entire operation, preventing a half-renamed inconsistent state |
| T-06-08 | Denial of Service | Multiple `PrismaClient` instances exhausting Postgres connection pool | mitigate | `server/src/db/prismaClient.js` is the single instantiation point; all services (this phase and future phases) import the shared singleton instead of creating their own client |
| T-06-09 | Tampering | Service layer bypassed by a future code path writing directly via Prisma without going through `kotaService.js` | accept | The Postgres-level FK constraint (created in Plan 01's migration) remains as defense-in-depth — a raw delete that skips this service layer still fails at the database, just with a less user-friendly error; acceptable since Phase 8 is the only consumer and will be required to route through this service layer by code review, not a runtime gate |
</threat_model>
