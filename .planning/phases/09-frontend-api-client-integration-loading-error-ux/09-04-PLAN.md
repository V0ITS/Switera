- GET /riwayat-keputusan → `[...]` (auth; root-mounted in index.js)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → /keputusan | Decision writes are role-gated; the status-change PUT is the one genuinely race-prone path (LOGIC-02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-D-RACE | Tampering / Integrity | Concurrent PUT /keputusan/:id status change | mitigate | The server's optimistic lock (LOGIC-02, 08-04) returns 409 to the losing request; the client does NOT optimistically mutate the cache before the response and surfaces the 409 as a visible Toast + keeps the modal open, so a double-allocation can never appear as success in the UI. |
| T-09-D-RBAC | Elevation of Privilege | POST/DELETE/restore vs PUT role sets | mitigate | Server enforces requireRole per route (POST/DELETE/restore: Admin+Manajer; PUT status: +Tim Logistik) per keputusanRoutes.js (AUTH-03); a disallowed token is rejected 403 and surfaced via runMutation, independent of UI gating. |
| T-09-D-401 | Spoofing | Expired token on any keputusan request | mitigate | Inherits 09-01's 401 handler — forces re-login, no silent failure. |
| T-09-D-STALE | Integrity | Stale decision/riwayat cache after cancel/restore | mitigate | removeKeputusan/restoreKeputusan refresh riwayat from the server (loadRiwayatKeputusan) rather than mirroring the server's "dibatalkan" transition client-side, eliminating divergence. |
| T-09-D-SC | Tampering | npm installs | mitigate | No new dependency; no install task; nothing to audit. |
</threat_model>
