- GET /permintaan → `[...]` (auth)

- DELETE /permintaan/:id (Admin) → `{ id }`

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → /permintaan | Request writes are Admin-only mutations validated server-side by Zod |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-P-RBAC | Elevation of Privilege | POST/PUT/DELETE /permintaan | mitigate | Server enforces requireRole("Admin") (permintaanRoutes.js); a non-Admin token is rejected 403 regardless of UI gating (AUTH-03), surfaced as a Toast via runMutation. |
| T-09-P-VAL | Tampering | Invalid input (negative tonase, empty date) | mitigate | Server-side Zod (permintaanSchemas.js) rejects with 400 + field-level messages (API-02); the client surfaces them via the thrown error's .fields and the runMutation Toast — never a silent accept. |
| T-09-P-401 | Spoofing | Expired token on any permintaan request | mitigate | Inherits 09-01's 401 handler — forces re-login, no silent failure. |
| T-09-P-DUP | Tampering / Integrity | Duplicate-request bypass | mitigate | The duplicate check is answered by the server (GET /permintaan/duplikat) against live DB state, not a possibly-stale client cache, closing the client/server divergence window. |
| T-09-P-SC | Tampering | npm installs | mitigate | No new dependency; no install task; nothing to audit. |
</threat_model>
