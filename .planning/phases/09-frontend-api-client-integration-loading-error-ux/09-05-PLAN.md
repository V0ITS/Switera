<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → /notifikasi, /activity-log | Notification reads/marks are per-authenticated-user; activity-log read is Admin-only |
| unauthenticated visitor → Landing | The public marketing page must render without a token and must not call authed endpoints |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-N-INTEGRITY | Tampering | Client-fabricated notification / activity-log | mitigate | tambahNotifikasi/catatAktivitas are converted to throwing shims — no client POST route exists (LOGIC-03); only server mutations generate notifications/log entries, so a client cannot fabricate history or notifications. |
| T-09-L-RBAC | Information Disclosure / Elevation | GET /activity-log on non-Admin | mitigate | Server enforces requireRole("Admin") (activityLogRoutes.js); loadActivityLog quietly degrades a 403 to an empty list rather than leaking an error, and the page is Admin-only in the menu anyway (AUTH-03 is the real boundary). |
| T-09-LAND-401 | Availability | Landing calling authed endpoints pre-login | mitigate | Landing is kept fully static (seed/inline demo data) and makes zero API calls, so an unauthenticated visitor never triggers a 401 redirect loop or a broken marketing page. |
| T-09-H-401 | Spoofing | hydrate() with no/expired token | mitigate | hydrate() no-ops without a token; an expired token during hydrate trips the 09-01 401 handler, clearing the session and forcing re-login rather than rendering stale data. |
| T-09-N-SC | Tampering | npm installs | mitigate | No new dependency; no install task; nothing to audit. |
</threat_model>
