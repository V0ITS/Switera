## THE CENTRAL ARCHITECTURAL DECISION (binding for all of Phase 9)

Today every page reads state synchronously via `useState(store.getState())` and re-renders through `store.subscribe(setSnapshot)` (verified: App.jsx:61/86, Layout.jsx:214/237, and all 12 pages). Mutations are called synchronously in handlers (e.g. `store.addPermintaan(...)`). The public method NAMES and the `subscribe`/`notify` contract MUST NOT change (ROADMAP Phase 9 Success Criterion 1).

**Chosen pattern: hydrated in-memory cache.**

1. `store.js` keeps its existing in-memory `state` object (same shape it has today) as a CACHE. All synchronous getters (`getState`, `getDaftarKota`, `getPermintaan`, `getKeputusan`, `getNotifikasi`, `getStokTbs`, etc.) keep returning from this cache synchronously — so NO page-level read code changes, and the "instant reflect, no refresh" feel via `notify()` is preserved unchanged.
2. Mutator methods change internally from "write cache + persist localStorage + notify" to "`await apiFetch(...)` against the REST API → write the server's authoritative response into the cache → `notify()`". Each mutator now returns a Promise.
3. **Call-site contract:** mutators return Promises. Pages that need to act after a mutation completes (close a modal, show success Toast) add a minimal `await` in their existing async handler — this is the default and is required where the v1.0 code already runs follow-up code after the call (e.g. ManajemenKota.submitForm shows a success Toast right after `store.tambahKota`). Pages that were pure fire-and-forget keep working because the post-fetch `notify()` still drives the re-render. This is the same approach 08-04 took for documenting LOGIC-02 instead of leaving it a TODO: it is a decision, not a footnote.
4. **Bootstrap hydration:** because reads are synchronous against the cache, the cache must be filled from the server before/at the moment an authenticated page renders. `store.hydrate()` (introduced here, fully wired in 09-05) fetches the server collections and populates the cache, then `notify()`. 09-01 introduces the function and calls it on successful login; 09-05 completes the full collection set and the app-init re-hydrate path.
5. **localStorage role changes:** the server is now the source of truth for all DOMAIN data. `persistState()` stops writing domain collections. Only three things stay in localStorage: the JWT (`switera_token`), the active user (`switera_user`, so a refresh keeps the session), and `tema` (UI preference, never a backend concern). The old `switera_state_v1` blob is no longer written; domain data always comes from the API.

## JWT STORAGE DECISION (binding, explicit — not silently assumed)

The JWT is stored in `window.localStorage` under key `switera_token`, consistent with this project's existing plaintext-localStorage session pattern (v1.0 stored `userAktif` in localStorage in plaintext; PROJECT.md and REQUIREMENTS.md never made secure session storage a v1.0/v2.0 requirement, and AUTH-05/06 refresh-token/httpOnly-cookie work is explicitly deferred to v2 in REQUIREMENTS.md). The XSS-exposure tradeoff of localStorage vs in-memory is ACCEPTED for this school-demo milestone and recorded in the threat model below (T-09-JWT). `apiClient` reads this key and attaches `Authorization: Bearer <token>` on every request.

## LOADING / ERROR UX DECISION (binding — FE-02)

`apiClient.apiFetch` increments a module-level in-flight counter before the request and decrements it after (success or failure); `store` exposes the derived boolean on its cache as `state.isLoading` (true when counter > 0) and the last error as `state.lastError`, both surfaced through the EXISTING `getState()`/`subscribe` contract so any page can read `snapshot.isLoading` with zero new wiring. On a failed request (network error or non-2xx), `apiFetch` throws a normalized error carrying the server's Indonesian `error`/`fields` message; `store` mutators catch it, set `state.lastError`, route the message through the existing `showToast({ type: "error", ... })` from `src/components/Toast.jsx`, `notify()`, and re-throw so the calling page's existing `try/catch` (e.g. ManajemenKota.submitForm) still sets field errors exactly as in v1.0. Every later plan reuses this mechanism — no plan reinvents loading/error handling.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-JWT | Information Disclosure | JWT in localStorage (apiClient setToken) | accept | Stored under `switera_token` in localStorage, consistent with the project's existing plaintext-localStorage session pattern. | T-09-401 | Spoofing / Elevation | Stale authenticated UI after token expiry | mitigate | apiClient detects any 401, calls clearToken() and the store's onUnauthorized handler which nulls userAktif + notifies; App.jsx's existing `!snapshot.userAktif` effect redirects to login. No silent retry, no stale authenticated-looking screen (AUTH-04 frontend half). |
| T-09-ATTACH | Spoofing | Missing/incorrect Authorization header | mitigate | apiClient is the single code path for all authenticated requests and unconditionally attaches `Authorization: Bearer <token>` when a token exists; no page builds its own fetch, so a request can never accidentally omit the header. |
| T-09-ENUM | Information Disclosure | Login error messaging | accept | The server returns one generic 401 for unknown-user/wrong-password/wrong-role (T-07-ENUM). Login.jsx no longer inspects credentials client-side to produce a more specific message — the v1.0 per-field breakdown is intentionally dropped to avoid re-introducing a client-side enumeration oracle. |
| T-09-SC | Tampering | npm installs | mitigate | This plan introduces NO new npm dependency (uses the browser-native fetch). No package install task exists; nothing to audit. |
