<objective>
Add server-side authorization on top of Plan 01's authentication: a requireAuth middleware that rejects missing/expired/malformed JWTs with 401 (AUTH-04), and a requireRole middleware that rejects a valid-but-wrong-role request with 403 (AUTH-03). Prove both with demo protected routes, establishing the middleware contract that every Phase 8 mutating route will reuse.

Purpose: AUTH-03/AUTH-04 require that authorization is enforced at the server, independent of what the client UI shows or hides (src/utils/navigation.js is cosmetic only and is NOT a security boundary). This plan creates the reusable guard middleware so Phase 8's CRUD endpoints inherit RBAC by composition rather than re-implementing it per route.
Output: server/src/middleware/requireAuth.js, server/src/middleware/requireRole.js, demo protected routes wired into the existing app, and a standalone RBAC verification script proving 401/403/200.
</objective>

<artifacts_produced>
Files/symbols this plan creates (the RBAC contract Phase 8 routes reuse):
- `server/src/middleware/requireAuth.js` — default-exported Express middleware. Reads the `Authorization: Bearer <token>` header; on missing/empty header responds 401 `{ error: "Token tidak ada." }`; calls `verifyToken`; on any failure (expired, malformed, bad signature, wrong alg) responds 401 `{ error: "Token tidak valid atau kedaluwarsa." }`; on success sets `req.user = { id, username, role }` and calls `next()`.
- `server/src/middleware/requireRole.js` — default-exported factory `requireRole(...allowedRoles)` returning a middleware that responds 403 `{ error: "Anda tidak memiliki izin untuk aksi ini." }` when `req.user.role` is not in `allowedRoles`, else `next()`. Assumes requireAuth ran first.
- `server/src/routes/protectedRoutes.js` — demo router proving the guards: e.g. `GET /protected/me` (any authenticated role), `POST /protected/admin-only` (requireRole("Admin")), `POST /protected/manajer-only` (requireRole("Manajer Distribusi")). These exist to satisfy AUTH-03/04 acceptance now; Phase 8 replaces them with real domain routes using the same middleware.
- `server/src/middleware/rbac.verify.mjs` — standalone script proving the 401 (no/expired/malformed token), 403 (wrong role), and 200 (right role) paths against the live app.
</artifacts_produced>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → protected route | Untrusted (possibly forged/expired) JWT presented for a privileged action |
| JWT payload → req.user → requireRole | The role claim drives the authorization decision; it must come from a server-signed token, never a client header/body |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-401 | Spoofing | server/src/middleware/requireAuth.js | mitigate | Every protected route requires a valid Bearer token; missing/expired/malformed/tampered tokens are rejected with 401, never silently allowed or retried — leverages jwt.verify's HS256 pinning from Plan 01 (T-07-ALG) |
| T-07-RBAC-BYPASS | Elevation of Privilege | server/src/middleware/requireRole.js + protectedRoutes.js | mitigate | Authorization is server-side middleware applied per route with an explicit role allow-list; it does NOT consult the frontend menuByRole (cosmetic only). A role hidden in the UI is still rejected at the API with 403. Verified by rbac.verify.mjs's wrong-role 403 assertion. |
| T-07-RBAC-OMIT | Elevation of Privilege | server/src/routes/protectedRoutes.js (and future Phase 8 routes) | mitigate | requireRole is applied AFTER requireAuth on each mutating route; the demo routes establish the mandatory ordering pattern. Risk that a future route omits the guard is mitigated by making the middleware composition the documented, copy-paste contract for Phase 8; the verify script is reusable as a regression check. |
| T-07-ROLE-FORGE | Spoofing | server/src/middleware/requireAuth.js | mitigate | req.user.role is read ONLY from the verified JWT payload, never from a client-supplied header/body field — a client cannot inject a higher role into the authorization decision |
| T-07-ERRLEAK2 | Information Disclosure | server/src/middleware/requireAuth.js | mitigate | The raw jwt verification error (expired vs malformed vs bad-signature) is not echoed to the client; a single generic 401 message is returned |
</threat_model>
