<objective>
Stand up the first real Express HTTP server for Switera and implement server-side authentication: a login route that bcrypt-verifies credentials and issues a signed HS256 JWT (AUTH-01), and a register route that stores a bcrypt-hashed password (AUTH-02), replacing the client-only plaintext auth in src/store.js.

Purpose: Phase 6 built the database layer but no HTTP server exists yet. This plan creates the Express app entrypoint that all of v2.0's API surface (Phases 7-10) builds on, plus the credential-verification and account-creation backbone every later route depends on.
Output: server/src/index.js (app entrypoint), server/src/auth/jwt.js (token signing), server/src/services/akunService.js (bcrypt auth + register), server/src/routes/authRoutes.js (POST /auth/login, POST /auth/register), and a standalone verification script.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → /auth/login | Untrusted username/password crosses here; credential-stuffing / brute-force exposure |
| client → /auth/register | Untrusted account fields (nama, username, role) crosses here |
| process env → jwt.js | JWT_SECRET enters the trust base; a weak/committed secret forges any token |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-SECRET | Information Disclosure / Spoofing | server/src/auth/jwt.js | mitigate | JWT_SECRET read only from process.env, never hardcoded; signToken throws if missing or still the placeholder string; server/.env is gitignored (Phase 6) so the secret is never committed |
| T-07-ALG | Spoofing | server/src/auth/jwt.js | mitigate | jwt.verify pinned to { algorithms: ["HS256"] }; tokens with alg none or any other algorithm are rejected — blocks algorithm-confusion forgery |
| T-07-PWHASH | Information Disclosure | server/src/services/akunService.js | mitigate | Passwords hashed with bcrypt cost factor 10 (matching seed.js); plaintext exists only as a transient function argument, never written to the DB; login uses bcrypt.compare, never plaintext equality |
| T-07-ENUM | Information Disclosure | server/src/routes/authRoutes.js (login) | mitigate | Login returns the same generic 401 "Username atau password salah." for both unknown-user and wrong-password, so an attacker cannot enumerate valid usernames |
| T-07-ROLESPOOF | Elevation of Privilege | server/src/routes/authRoutes.js (login) | mitigate | Login verifies username+password only and derives role from the stored account, NOT from a client-supplied role field (the v1 client passed role into cariAkun) — a client cannot self-assign a higher role at login |
| T-07-BRUTE | Denial of Service / Spoofing | server/src/routes/authRoutes.js (login) | accept | No rate-limiting on the login route this milestone. Accepted: 3-account, single-process, local-only school demo with no public exposure. Documented as a v2 candidate (would add express-rate-limit) — explicitly out of scope, not silently omitted. |
| T-07-ERRLEAK | Information Disclosure | server/src/routes/authRoutes.js | mitigate | Handlers catch errors and return generic Indonesian messages; raw Prisma/stack-trace details are never sent to the client |
</threat_model>
